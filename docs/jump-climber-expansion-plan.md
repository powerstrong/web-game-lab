# Jump Climber Expansion Plan

`jump-climber`를 단일 캐릭터 솔로 점프 프로토타입에서
대표 타이틀 수준의 확장 가능한 게임으로 키우기 위한 작업 계획이다.

## Goal

이번 확장의 핵심 목표는 아래 네 가지를 하나의 일관된 사용자 경험으로 묶는 것이다.

1. 플레이 전에 3개의 캐릭터 중 하나를 고를 수 있어야 한다.
2. 원하면 사용자 얼굴 사진을 캐릭터 얼굴 영역에 합성해 프리뷰할 수 있어야 한다.
3. 2인 동시 플레이가 가능해야 한다.
4. 모바일에서는 화면 전체를 좌/우 영역으로 나눠 입력하는 방식으로 전환해야 한다.

## Product Direction

게임 톤은 "짧게 켜도 바로 귀엽고 재밌는 점프 게임"이다.
이번 작업은 단순히 기능을 더하는 것이 아니라, 아래 경험을 만드는 데 목적이 있다.

- 캐릭터를 고르는 순간부터 애착이 생긴다.
- 얼굴 커스터마이즈가 장난감처럼 가볍고 즉시 반응한다.
- 2인 플레이가 시작되면 같은 화면을 공유하는 재미가 생긴다.
- 모바일에서도 조작 설명 없이 바로 플레이할 수 있다.

## Feature Breakdown

### 1. Character Selection

출시 1차 목표는 기본 캐릭터 3종 선택이다.

권장 방향:

- `모찌 토끼`
- `푸딩 햄스터`
- `아기 병아리`

요구사항:

- 시작 전 선택 화면에서 3개 캐릭터를 모두 미리 볼 수 있어야 한다.
- 선택된 캐릭터의 이름과 대표 포즈가 보여야 한다.
- 1P와 2P가 서로 다른 캐릭터를 고를 수 있어야 한다.
- 같은 캐릭터를 중복 선택할지 여부는 구현 시점에 결정하되, 문서상 기본값은 "중복 허용"으로 둔다.

산출물:

- 캐릭터 카드 UI
- 선택 상태 표시
- 선택 결과를 게임 시작 데이터에 반영하는 구조

### 2. Face Swap Customization

사용자가 자기 얼굴 이미지를 넣어 캐릭터 얼굴 부분과 교체할 수 있어야 한다.

요구사항:

- 사용자는 이미지 파일을 업로드할 수 있어야 한다.
- 업로드 직후 캐릭터 얼굴 영역에 적용된 미리보기를 볼 수 있어야 한다.
- 얼굴 위치와 크기를 간단히 조절할 수 있어야 한다.
- 1P와 2P는 각자 별도 프리셋을 가질 수 있어야 한다.
- 얼굴 합성을 쓰지 않는 기본 캐릭터 상태로 언제든 돌아갈 수 있어야 한다.

추천 구현 원칙:

- 첫 버전은 브라우저 로컬 처리만 사용한다.
- 얼굴 검출 같은 무거운 자동화 대신, 원형 마스크 + 이동/확대 조절부터 시작한다.
- 최종 게임 플레이에서는 프리뷰에서 확정한 얼굴 이미지를 그대로 사용한다.

필요 UI:

- 업로드 버튼
- 얼굴 적용 on/off 토글
- 확대/축소 슬라이더
- 좌우/상하 위치 조절
- 실시간 프리뷰 패널

### 3. Motion Variants

캐릭터는 좌우 이동에 따라 일부 모션이 달라져야 한다.

최소 요구사항:

- 정면 또는 기본 점프 포즈
- 왼쪽 이동 포즈
- 오른쪽 이동 포즈
- 상승 중 / 하강 중 차이를 줄 수 있으면 반영

구현 원칙:

- 처음부터 프레임 애니메이션보다 "상태 기반 이미지 교체"를 우선한다.
- `idle/jump-left/jump-right/fall` 정도의 명확한 상태 분리가 적당하다.
- 3개 캐릭터 모두 동일한 상태 이름 체계를 공유해야 한다.

예시 상태:

- `jump_neutral`
- `jump_left`
- `jump_right`
- `fall_neutral`

### 4. Two-Player Gameplay

2인 플레이는 같은 월드를 공유하는 로컬 동시 플레이를 1차 목표로 한다.

요구사항:

- 1P와 2P가 동시에 움직일 수 있어야 한다.
- 두 플레이어는 각각 독립적인 위치, 속도, 생존 상태를 가진다.
- 둘 중 한 명이 죽어도 남은 한 명은 계속 플레이한다.
- 게임 오버는 두 플레이어가 모두 탈락했을 때만 발생한다.

카메라 규칙:

- 카메라는 더 높은 플레이어가 아니라 "더 낮은 플레이어" 기준으로 이동한다.
- 이유는 뒤처진 플레이어를 화면 밖으로 덜 잃게 하기 위해서다.
- 다만 너무 낮은 플레이어 때문에 상단 플레이가 답답해지지 않도록 추적 완충값을 둔다.

기록 규칙:

- HUD에 `1P Best`, `2P Best`를 각각 표시한다.
- 해당 플레이어가 사망한 뒤에도 최고 기록은 유지해서 보여준다.
- 최종 결과 화면이 생기면 플레이어별 최고 높이를 함께 보여준다.

### 5. Mobile Input Redesign

모바일 조작은 기존 하단 좌/우 버튼 대신 맵 전체를 좌/우 반으로 나눈 터치 입력으로 바꾼다.

요구사항:

- 화면 왼쪽 절반 터치 시 왼쪽 이동
- 화면 오른쪽 절반 터치 시 오른쪽 이동
- HUD나 메뉴를 누를 때는 게임 입력으로 오작동하지 않게 처리
- 멀티터치 시 1P/2P 동시 입력 규칙을 문서로 먼저 정리

기본 제안:

- 솔로 플레이에서는 첫 터치 기준으로 좌/우 이동
- 2인 플레이에서는 한 손가락씩 동시에 입력 가능하도록 다중 포인터를 지원
- 터치 시작 영역 기준으로 방향을 유지하고, 떼면 해제

검토 포인트:

- 2인 로컬 플레이 모바일 버전의 입력 충돌
- 두 손가락이 모두 같은 반을 눌렀을 때의 처리
- 선택 화면과 실제 게임 화면의 터치 역할 분리

## UX Flow

권장 화면 흐름:

1. 모드 선택
2. 1P / 2P 선택
3. 캐릭터 선택
4. 얼굴 커스터마이즈 선택 사항
5. 프리뷰 확인
6. 게임 시작
7. 플레이 중 개별 최고 높이 확인
8. 두 플레이어 모두 탈락 시 결과 표시

첫 버전에서는 3, 4, 5를 하나의 사전 설정 패널로 합쳐도 된다.

## Technical Plan

### Phase 1. Pre-Game Configuration Layer

게임 시작 전에 필요한 설정 상태를 별도 구조로 분리한다.

필요 상태:

- player count
- selected character per player
- face swap enabled per player
- uploaded face asset per player
- face transform per player

이 단계에서 해야 할 일:

- 시작 화면 UI 정의
- 설정 상태와 게임 상태 분리
- 재시작 시 설정 유지 여부 결정

### Phase 2. Character Asset Contract

캐릭터 아트를 코드가 다루기 쉬운 규칙으로 고정한다.

권장 파일 규칙:

- `character/<id>/jump_neutral.png`
- `character/<id>/jump_left.png`
- `character/<id>/jump_right.png`
- `character/<id>/fall_neutral.png`
- `character/<id>/face_mask.png` 또는 얼굴 앵커 정보

추가 데이터:

- 얼굴 중심 좌표
- 권장 얼굴 크기
- 발판 충돌 기준용 발 위치

### Phase 3. Gameplay Refactor For Multiple Players

현재 단일 `state.player` 구조를 배열 또는 슬롯 기반 구조로 바꾼다.

필요 변경:

- `players[]` 구조로 확장
- 플레이어별 입력 상태
- 플레이어별 충돌 판정
- 플레이어별 최고 높이 계산
- 생존 플레이어 수 기준 게임 종료 판정

### Phase 4. Camera Rules

카메라 계산을 단일 플레이어 추적에서 다중 플레이어 추적으로 전환한다.

기본 규칙:

- 살아있는 플레이어 중 가장 낮은 위치를 기준점으로 삼는다.
- 급격한 흔들림을 줄이기 위해 lerp 또는 완충값을 유지한다.
- 화면 밖 여유 영역을 둬서 둘 다 너무 답답하지 않게 한다.

### Phase 5. Input Abstraction

입력 시스템을 키보드와 터치 공통 레이어로 정리한다.

필요 입력 소스:

- keyboard 1P
- keyboard 2P
- touch area left/right
- pointer tracking for multi-touch

키보드 제안:

- 1P: `A` / `D`
- 2P: `ArrowLeft` / `ArrowRight`

이 매핑은 구현 시 문서와 UI 설명에 함께 반영한다.

### Phase 6. HUD and Results

HUD는 단순 높이 표시에서 플레이어별 정보를 가진 형태로 바뀌어야 한다.

최소 표시:

- `1P Best`
- `2P Best`
- 현재 생존 상태
- 재시작 버튼

추가 고려:

- 1P / 2P 캐릭터 초상
- 현재 선택 캐릭터 이름
- 얼굴 커스터마이즈 여부 표시

## Risks

### 1. Scope Risk

캐릭터 선택, 얼굴 합성, 2인 플레이, 모바일 입력 개편은 각각만으로도 큰 작업이다.
한 번에 구현하면 UI와 게임 로직이 동시에 흔들릴 수 있다.

대응:

- 프리게임 설정 레이어를 먼저 만들고
- 그 다음 아트 계약
- 그 다음 2인 플레이
- 마지막에 입력 세부 튜닝 순서로 간다

### 2. Mobile Two-Player Complexity

한 화면에서 2인이 동시에 터치하는 경우 오입력과 포인터 충돌 가능성이 높다.

대응:

- 1차 버전은 로컬 2인 키보드 플레이를 먼저 안정화
- 모바일 2인 터치는 후속 단계에서 검증

### 3. Face Swap Quality Variance

사용자 얼굴 사진 비율과 구도가 제각각이라 결과물이 쉽게 어색해질 수 있다.

대응:

- 자동 보정보다 수동 위치/크기 조절 우선
- 프리뷰를 크게 보여서 시작 전에 확인 가능하게 함

1. 설정 화면과 캐릭터 3종 선택 UI

3. 얼굴 업로드 + 프리뷰

7. 키보드 2인 플레이 완성
8. 모바일 전체 화면 좌/우 입력 적용
9. 모바일 2인 멀티터치 튜닝

## Implementation Notes For The Next Coding Pass

다음 구현 단계에서는 아래를 먼저 확인한다.

- 현재 `game.js`에서 단일 플레이어 상태가 어느 함수에 강하게 결합되어 있는지
- 프리게임 설정 UI를 `index.html` 안에 둘지 오버레이로 둘지
- 캐릭터 아트 로딩 방식을 CSS background로 할지 `img` / `canvas`로 할지
- 얼굴 프리뷰를 DOM 합성으로 시작할지 `canvas` 합성으로 시작할지

이 문서는 구현 전에 업데이트하는 기준 문서이며,
구현 도중 범위나 우선순위가 바뀌면 먼저 이 문서를 수정한다.
