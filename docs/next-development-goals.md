# 다음 개발 목표 정리

## 문서 목적

이 문서는 현재 코드베이스를 실제로 분석한 뒤, 이미 구현된 싱글플레이 중심 허브와 로비 구조를 버리지 않고 어떻게 확장할지 정리한 문서다.

이번 단계의 중심은 다음이다.

- 새로운 프로젝트를 처음부터 다시 설계하지 않는다.
- 현재 허브, 로비, 프로토타입 구조를 최대한 재사용한다.
- 실제 대규모 멀티플레이 구현보다 먼저 게임 분류, 메타정보, 목록 UX, 입력 레이어 기준을 정리한다.
- 긴 소개 문구보다 게임 카드와 메타정보 중심 구조를 우선한다.

## 현재 구현 상태 요약

### 1. 허브/진입 화면

현재 루트 화면은 이미 게임 허브의 시작점 역할을 하고 있다.

- 루트 `index.html`에는 메인 랜딩과 함께 게임 바로가기 2개가 노출된다.
- 현재 구조는 "메타정보 중심 게임 허브"라기보다 "소개형 랜딩 + 프로토타입 바로가기"에 가깝다.
- 게임 카드 영역은 이미 존재하지만, 카드에 필요한 공통 메타정보는 제목과 짧은 설명 정도만 노출된다.
- CTA는 `lobby/index.html`로 연결되어 멀티 대기실 흐름으로 진입할 수 있다.

정리하면, 허브 자체는 이미 있지만 아직은 데이터 기반 목록 페이지가 아니라 정적 소개 화면에 가깝다.

### 2. 로비/대기실 구조

`lobby/`는 이미 멀티플레이 전 단계의 기본 흐름을 갖추고 있다.

- 닉네임 입력
- 4자리 방 코드 생성/입장
- 세션 기준 닉네임 저장
- 방 코드 복사
- 참가자 목록 표시
- 게임 투표
- 준비 완료 투표
- 간단한 채팅
- 연결 상태 표시 및 재연결 오버레이
- 카운트다운 후 게임 페이지 이동

즉, "같이 모여서 어떤 게임을 시작할지 정하는 공간"은 이미 구현되어 있다.

### 3. 게임 구현 상태

현재 `prototypes/` 아래에는 최소 2개의 플레이 가능한 게임과 1개의 템플릿이 있다.

- `dodge-square`
- `rhythm-tap`
- `_template`

두 게임 모두 공통적으로 다음 특성을 가진다.

- 프레임워크 없이 HTML/CSS/JS로 독립 실행된다.
- 한 폴더 안에 `index.html`, `style.css`, `game.js`가 모여 있다.
- 재시작 버튼과 짧은 판 구조를 가진다.
- 현재는 로컬 단일 화면 플레이 기준이다.
- 게임 상태와 입력 처리가 각 파일 내부에 직접 묶여 있다.

현재 구현만 기준으로 보면 두 게임은 아직 `SOLO`에 가깝다. 로비에서 게임으로 이동하는 흐름은 있으나, 실제 게임 내부에 멀티 판정이나 결과 제출 구조는 아직 없다.

### 4. 서버/실시간 구조

`worker/`에는 Cloudflare Worker + Durable Object 기반의 방 관리 코드가 이미 있다.

- 방 생성 API
- 방별 WebSocket 연결
- 플레이어 입장/퇴장 관리
- 채팅 로그 저장
- 게임 투표 집계
- 준비 완료 집계
- 과반 이상일 때 게임 시작 브로드캐스트

즉, 실시간 로비 인프라는 이미 있고, 이것은 이후 `DUEL_LIVE`와 `PARTY_ASYNC` 확장 시 그대로 활용 가능한 기반이다.

## 현재 구조에서 재사용 가능한 부분

### 1. 게임 폴더 단위 구조

`prototypes/<game>/index.html + style.css + game.js` 구조는 유지 가치가 높다.

- 미니게임 단위가 분리되어 있다.
- 실험/추가/삭제가 쉽다.
- 짧은 게임을 빠르게 붙이기에 적합하다.
- `_template`이 이미 있어 새 게임 추가 진입 비용이 낮다.

이 구조는 이후에도 유지하되, 게임 등록 정보만 바깥 공통 레이어로 분리하면 된다.

### 2. 현재 카드형 UI 방향

루트와 로비 모두 카드형 UI, 짧은 문구, 빠른 액션 버튼 중심으로 구성되어 있다.

- 큰 설명보다 클릭 가능한 항목이 먼저 보인다.
- 모바일 폭에서도 크게 무너지지 않는 반응형 레이아웃이 이미 있다.
- 시각 톤이 허브/로비/게임 사이에서 어느 정도 일관된다.

이 방향은 유지하고, 텍스트를 늘리는 대신 카드 메타정보를 강화하는 쪽으로 가는 것이 맞다.

### 3. 로비 진입 및 세션 처리

현재 로비는 향후 멀티 타입 확장 시 그대로 재사용 가능하다.

- 닉네임 저장
- 방 코드 기반 진입
- 방 코드 복사
- 연결 상태 표시
- 재연결 처리

특히 `PARTY_ASYNC`에서도 "방 입장 -> 게임 선택/시작 -> 결과 제출/비교"의 앞부분은 같은 구조를 재사용할 수 있다.

### 4. 실시간 상태 동기화 기본 골격

`worker/src/room.js`의 메시지 흐름은 이미 확장 가능한 형태다.

- `join`
- `chat`
- `vote_game`
- `vote_start`
- `game_start`

지금은 메시지 종류가 적지만, 이후 `submit_result`, `round_end`, `scoreboard`, `rematch_vote` 같은 메시지를 붙이기 좋은 출발점이다.

## 현재 구조의 한계와 확장이 필요한 영역

### 1. 게임 메타정보가 중앙화되어 있지 않다

현재 게임 정보가 여러 곳에 중복 정의되어 있다.

- 루트 `index.html`의 게임 스트립
- `lobby/room.html`의 게임 카드 마크업
- `lobby/room.js`의 `GAME_META`
- `worker/src/room.js`의 `GAME_PATHS`

이 상태에서는 게임이 늘어날수록 제목, 설명, 경로, 타입이 서로 어긋날 가능성이 높다.

가장 먼저 필요한 일은 "게임 공통 레지스트리"를 만드는 것이다.

### 2. 현재 허브는 싱글플레이 목록 허브로 정리되어 있지 않다

루트 화면에 게임 링크는 있지만, 아직 허브가 아래 질문에 답하지 못한다.

- 이 게임은 혼자 하는가, 같이 하는가
- 권장 인원은 몇 명인가
- 실제 플레이 가능 인원은 몇 명인가
- 한 판은 얼마나 걸리는가
- 지금 플레이 가능한 상태인가

즉, 현재 허브는 "게임을 소개하는 화면"이지 "게임을 분류해서 고르는 화면"은 아니다.

### 3. 게임과 로비 사이 공통 계약이 없다

로비는 `code`와 `name`을 쿼리로 넘기지만, 현재 프로토타입 게임은 이를 사용하지 않는다.

즉, 현재는 "로비에서 시작은 시키지만 게임은 각자 로컬에서 독립 실행" 상태다.

이 문제는 당장 대규모 멀티를 구현하자는 뜻이 아니라, 최소한 아래 계약이 필요하다는 뜻이다.

- 게임 ID
- 게임 타입
- 입력 모드
- 방 코드/플레이어 정보 사용 여부
- 점수/기록 결과 반환 형식

### 4. 입력 처리 공통 레이어가 없다

현재 입력은 각 게임 내부에 직접 박혀 있다.

- `dodge-square`: 방향키
- `rhythm-tap`: 스페이스
- 공통 터치 입력 없음
- 공통 입력 모드 저장 없음
- 자동 감지/수동 전환 없음

지금 방향대로 확장하려면 게임 로직과 입력 어댑터를 분리하는 공통 레이어가 필요하다.

### 5. PWA 준비 구조가 아직 없다

현재 코드베이스에는 다음이 없다.

- `manifest.webmanifest`
- service worker
- 설치 버튼 처리
- 오프라인/캐시 전략

이번 단계에서 바로 붙일 필요는 없지만, 허브 진입 구조와 자산 구조를 PWA 친화적으로 정리해둘 필요는 있다.

### 6. 테스트/검증 체계가 거의 없다

현재는 프로토타입 중심 구조라 자동화된 테스트나 공통 검증 계약이 없다.

이후 게임 수가 늘어나면 최소한 다음 정도는 필요하다.

- 게임 메타정보 누락 검사
- 허브 카드 렌더링 누락 검사
- 지원 인원/권장 인원 표기 검사
- 타입별 진입 경로 검사

## 게임 타입 분류 체계 제안

현재 코드베이스에 녹여 넣기 쉬운 분류는 아래와 같다.

| 타입 | 의미 | 현재 구조와의 연결 방식 |
| --- | --- | --- |
| `SOLO` | 혼자 플레이 중심, 개인 기록 저장 가능 | 현재 `prototypes/*` 구조를 가장 그대로 활용 가능 |
| `DUEL_LIVE` | 1~2인 실시간 대전/협동 | 현재 `lobby/`와 `worker/`가 가장 먼저 재사용될 영역 |
| `PARTY_ASYNC` | 3인 이상, 제한 시간 플레이 후 결과 제출/비교 | 현재 로비 구조를 유지하면서 게임 종료 후 결과 제출 단계만 추가하기 좋음 |

### 현재 게임에 대한 1차 분류

현재 구현 상태만 기준으로 하면 다음처럼 두는 것이 안전하다.

| 게임 ID | 현재 분류 | 비고 |
| --- | --- | --- |
| `dodge-square` | `SOLO` | 현재는 로컬 생존 기록형 |
| `rhythm-tap` | `SOLO` | 현재는 로컬 점수 기록형 |

향후 확장 방향은 아래처럼 열어둘 수 있다.

- `rhythm-tap`은 결과 제출형으로 바꾸면 `PARTY_ASYNC` 확장 후보가 될 수 있다.
- 반응속도/회피 계열 신규 게임은 `DUEL_LIVE`로 붙이기 좋다.

핵심은 "현재 구현 상태의 분류"와 "향후 확장 가능성"을 구분하는 것이다.

## 게임 메타정보 표준안

앞으로는 게임 등록 정보를 하나의 공통 데이터 소스로 관리하는 것이 좋다.

최소 필드는 다음을 권장한다.

| 필드 | 설명 |
| --- | --- |
| `id` | 내부 식별자 |
| `title` | 카드/상세에 노출할 게임명 |
| `description` | 한 줄 설명 |
| `type` | `SOLO`, `DUEL_LIVE`, `PARTY_ASYNC` |
| `recommendedPlayers` | 권장 인원 |
| `supportedPlayers` | 실제 지원 인원 |
| `playMode` | 예: `개인 기록`, `실시간 대전`, `기록 경쟁` |
| `durationSeconds` | 예상 플레이 시간 |
| `status` 또는 `badge` | 예: `PLAYABLE`, `ALPHA`, `COMING_SOON` |
| `thumbnail` | 카드 썸네일 경로 |
| `icon` | 작은 아이콘 또는 이니셜 |
| `accentColor` | 카드 강조색 |
| `path` | 실제 게임 진입 경로 |

예시 스키마:

```js
{
  id: "rhythm-tap",
  title: "리듬 탭 챌린지",
  description: "제한 시간 안에 최고 점수를 노리세요",
  type: "PARTY_ASYNC",
  recommendedPlayers: "3~4명",
  supportedPlayers: "1~4명",
  playMode: "기록 경쟁",
  durationSeconds: 30,
  status: "PLAYABLE",
  icon: "RT",
  accentColor: "#ffb845",
  path: "/prototypes/rhythm-tap/index.html"
}
```

### 중요한 표기 원칙

- `권장 인원`과 `플레이 가능 인원`은 반드시 분리해서 표기한다.
- 설명은 한 줄 이내를 유지한다.
- 카드에 들어갈 정보는 설명문보다 메타정보를 우선한다.

## 게임 목록/카드 UI 방향

### 유지할 것

- 첫 화면에서 바로 게임 목록이 보여야 한다.
- 카드 클릭 중심 흐름을 유지한다.
- 제목과 보조 문구는 짧게 유지한다.
- 모바일/데스크탑 공통으로 빠르게 훑히는 레이아웃을 유지한다.

### 바꿔야 할 것

현재 루트 화면은 소개 문구 비중이 크고 게임 메타정보가 약하다. 다음 방향으로 바꾸는 것이 적절하다.

- 히어로는 짧은 제목 1줄 + 보조 문구 1줄 수준으로 축소
- 그 아래에 실제 게임 목록 카드 배치
- 카드 안에 메타정보를 구조적으로 고정 표기
- 타입/상태 배지를 카드 상단에 표기
- 필터가 생긴다면 타입 기준 우선

권장 카드 정보:

- 게임명
- 한 줄 설명
- 권장 인원
- 플레이 가능 인원
- 플레이 방식
- 예상 플레이 시간
- 타입 배지 또는 상태 라벨

샘플 카드 정보 예시:

| 항목 | 예시 |
| --- | --- |
| 게임명 | 반응속도 배틀 |
| 설명 | 먼저 눌러 승부를 가르세요 |
| 권장 인원 | 2명 |
| 플레이 가능 | 1~2명 |
| 방식 | 실시간 대전 |
| 예상 시간 | 20초 |
| 타입 | `DUEL_LIVE` |

## 입력 방식 대응 방향

현재 코드는 게임별로 입력이 직접 구현되어 있다. 앞으로는 아래 원칙으로 공통화하는 것이 좋다.

### 원칙

- 기본은 자동 감지
- 잘못 감지될 수 있으므로 수동 전환 허용
- 마지막 선택은 저장 가능하도록 설계
- 게임 로직과 입력 처리 로직을 분리

### 현재 코드에 맞는 적용 방식

공통 입력 레이어를 하나 두고, 각 게임은 "입력 이벤트"만 받는 구조로 가는 것이 적절하다.

- `keyboard` 입력기
- `touch` 입력기
- `auto` 판별기
- 사용자 수동 전환 UI
- 마지막 선택 저장

예를 들어 각 게임은 아래 같은 형태를 공통으로 받는 쪽이 좋다.

```js
createGame({
  inputMode: "auto",
  onMove(direction) {},
  onTap() {},
  onPrimaryAction() {}
});
```

이렇게 하면 현재 `dodge-square`와 `rhythm-tap`의 게임 루프는 유지하면서도 입력 레이어만 갈아끼우기 쉬워진다.

## PWA 고려사항

현재는 아직 PWA 구현이 없지만, 이후 붙이기 쉬운 구조로 정리해야 한다.

정리 방향은 다음 정도면 충분하다.

- 허브가 앱 진입점 역할을 하도록 유지
- 게임 메타정보와 썸네일 경로를 정리해 manifest 연결을 쉽게 만들기
- 정적 파일 경로를 일관되게 관리하기
- 추후 설치 버튼을 허브 상단의 짧은 보조 액션으로 추가하기 쉽게 만들기

이번 단계에서 바로 구현할 필요는 없지만, 허브 구조를 다시 만들 때 PWA 진입점을 고려하지 않고 분산시키면 나중에 비용이 커진다.

## 우선순위별 개발 단계 제안

### 1단계. 게임 공통 레지스트리 도입

가장 먼저 해야 할 일이다.

- 게임 메타정보를 한 곳으로 모은다.
- 루트 허브, 로비, 워커가 같은 데이터를 참조하게 만든다.
- 하드코딩된 게임명/설명/경로 중복을 제거한다.

### 2단계. 허브를 메타정보 중심 목록 화면으로 정리

- 긴 소개 문구를 줄인다.
- 게임 카드 목록을 메인 콘텐츠로 올린다.
- 타입/권장 인원/지원 인원/방식/시간 표기를 붙인다.
- 현재 플레이 가능 여부를 카드 상태로 드러낸다.

### 3단계. 타입별 진입 규칙 정리

- `SOLO`: 허브에서 바로 게임 진입
- `DUEL_LIVE`: 로비 생성/입장 후 게임 진입
- `PARTY_ASYNC`: 로비 진입 후 제한 시간 플레이 및 결과 제출 흐름 준비

핵심은 게임마다 진입 버튼 동작이 달라질 수 있도록 허브 레벨에서 규칙을 갖는 것이다.

### 4단계. 게임 공통 부트스트랩 계약 만들기

- 게임 시작 시 공통으로 받는 파라미터 정의
- 결과 제출 형식 정의
- 개인 기록/방 플레이 구분 정의
- 게임 종료 후 허브 또는 결과 화면으로 복귀하는 기준 정의

### 5단계. 입력 공통 레이어 도입

- 자동 감지
- 수동 전환
- 마지막 선택 저장
- 게임 로직과 입력 처리 분리

### 6단계. `PARTY_ASYNC` 결과 흐름 설계

이 단계에서 처음으로 멀티 느낌이 본격적으로 확장된다.

- 제한 시간 시작 신호
- 플레이 종료
- 점수 제출
- 순위 비교
- 재도전 또는 다음 게임 이동

실시간 동기화보다 결과 제출 흐름을 먼저 정리하는 것이 현재 프로젝트 방향과 잘 맞다.

### 7단계. PWA 기본 대응

- manifest
- service worker
- 설치 UX
- 앱처럼 접근 가능한 진입 구조

## 바로 실행할 할 일 목록

- [ ] 게임 공통 레지스트리 파일 위치 결정
- [ ] 현재 2개 게임 메타정보를 공통 레지스트리로 옮길 필드 설계
- [ ] 루트 허브를 소개형 화면에서 카드 목록 중심 화면으로 정리
- [ ] 카드에 `권장 인원` / `플레이 가능 인원` / `방식` / `예상 시간` / `타입`을 노출
- [ ] `SOLO`, `DUEL_LIVE`, `PARTY_ASYNC` 타입별 진입 흐름 정의

- [ ] 워커의 `GAME_PATHS`도 공통 메타 기반으로 관리할 구조 정리
- [ ] 각 게임이 받을 공통 부트스트랩 파라미터 정의
- [ ] 입력 공통 레이어 초안 설계
- [ ] 터치/키보드 자동 감지 + 수동 전환 + 마지막 선택 저장 정책 정리

## 리스크 및 주의사항

### 1. 게임 등록 정보 중복

지금 상태에서 게임 수가 늘어나면 가장 먼저 깨질 부분이다. 제목, 설명, 경로, 타입이 파일마다 따로 놀게 될 가능성이 높다.

### 2. 현재 로비는 "시작 전"까지는 강하지만 "플레이 중/종료 후" 계약이 약하다

로비는 이미 준비되어 있지만, 실제 게임이 결과를 어떤 형식으로 돌려주는지 정의되어 있지 않다. 이 부분을 정하지 않으면 `PARTY_ASYNC` 확장이 어려워진다.

### 3. 현재 프로토타입은 입력과 로직이 강하게 결합되어 있다

게임 수가 늘기 전에 분리 원칙을 잡아두지 않으면 모바일 대응 비용이 빠르게 커진다.

### 4. 긴 소개 문구로 다시 회귀하지 않도록 주의

현재도 랜딩 성격이 남아 있기 때문에, 허브 확장 작업을 하면서 카드보다 설명문이 앞에 오는 구조로 되돌아가지 않도록 주의해야 한다.

## 결론

현재 프로젝트는 "아무 것도 없는 상태"가 아니라 다음 세 가지가 이미 있다.

- 싱글플레이 프로토타입을 빠르게 붙일 수 있는 게임 폴더 구조
- 같이 모일 수 있는 로비와 실시간 방 인프라
- 카드형 UI와 짧은 판 중심의 미니게임 톤

따라서 다음 개발 목표의 핵심은 새로운 거대한 시스템을 만드는 것이 아니라, 이미 있는 구조 위에 아래 세 가지를 먼저 얹는 것이다.

1. 게임 공통 메타정보 체계
2. 타입 기반 허브/진입 구조
3. 입력/결과 처리 공통 계약

이 순서로 가면 현재 싱글플레이 허브를 유지한 채 `SOLO -> DUEL_LIVE -> PARTY_ASYNC` 확장을 자연스럽게 이어갈 수 있다.
